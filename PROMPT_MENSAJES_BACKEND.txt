3. PROBLEMA ACTUAL EN EL FRONTEND
===============================================================================

El endpoint GET /api/owner/messages devuelve TODOS los mensajes de
outbound_messages sin filtrar. Esto incluye mensajes tipo `general` que son
auto-respuestas del bot del WhatsApp del negocio (bienvenidas, menús de
opciones, respuestas fallback).

En el frontend ya aplicamos un filtro cliente-side para excluir messageType
=== 'general' y mostrar solo notificaciones ligadas a citas. Pero hay tres
flujos de comunicación inbound del cliente que el owner NUNCA ve:

  (A) El cliente escribe por WhatsApp solicitando una cita (intent: booking)
  (B) El cliente escribe por WhatsApp para pedir hablar con una persona
      real — opción 5 del menú del bot (intent: human_help)
  (C) El cliente escribe por WhatsApp solicitando un cambio de cita
      (intent: reschedule o cancel)

Estos tres casos quedan registrados en inbound_messages con su intent
clasificado, pero el owner no tiene forma de verlos en la app.

Exposición de la tabla inbound_messages filtrable por intent.

    limit: registros por página (default: 20)

      ok: true,
      data: [
        {
          id: string,
          channel: string,           // 'whatsapp' u otro

REQUERIMIENTO A — Ampliar el diccionario de normalización de intents
-------------------------------------------------------------------------------

El RPC detect_inbound_intent() y/o la función normalize_inbound_text() deben
reconocer correctamente los siguientes patrones de mensajes reales que están
llegando al WhatsApp del negocio y que actualmente caen como `unknown` o se
clasifican con baja confidence:

### Grupo 1 — Saludos (intent: `greeting`)
Estos mensajes deben clasificarse como `greeting` con alta confidence.
El sistema YA DEBERÍA responderles automáticamente con un saludo y el menú.

  Ejemplos reales detectados:
    "Hola buen dia"
    "Como estas Jaque?"
    "Como estas?"
    "Que tal"
    "Buen dia"
    "Holaaa"
    "Buenos dias"
    "Buenas tardes"
    "Buenas noches"
    "Hola, como estan"
    "Hola! buenos dias"
    "Ey hola"
    "Hey"

  Patrones normalizados a agregar al diccionario:
    /^(hola|ola|buenas|buenos dias|buen dia|buenas tardes|buenas noches|que tal|como estas|como estan|como te va|ey|hey|oye)[\s!?.,]*/
    Variaciones con doble letra: "holaaa", "holaa", "buenass", etc.

### Grupo 2 — Preguntas generales / dudas (intent: `human_help` con confidence media)
Estos mensajes indican que el cliente necesita hablar con alguien, pero no
necesariamente son urgentes. El bot puede responder con el menú y opción de
contacto.

  Ejemplos reales:
    "Tengo dudas"
    "Tengo una pregunta"
    "Necesito información"
    "Necesito info"
    "Quiero saber algo"
    "Me puede ayudar"
    "Buenas, tengo una consulta"

  Recomendación: intent = `human_help`, confidence = 0.6-0.7, needs_human_review = false
  (el bot puede responder con el menú de opciones; no requiere atención humana)

### Grupo 3 — Solicitud explícita de hablar con persona (intent: `human_help` urgente)
Estos mensajes SÍ requieren atención humana real. needs_human_review = true.

  Ejemplos:
    "Hablar con el estudio"
    "Hablar con alguien"
    "Necesito hablar con una persona"
    "Me comunican con Jaquelina"
    "Quiero hablar con el dueño"
    "Necesito que me llamen"
    "Opcion 5" (o simplemente "5" si es la opción del menú)

REQUERIMIENTO B — Nuevo campo `bot_replied` en inbound_messages
-------------------------------------------------------------------------------

### Problema detectado en frontend:

El frontend muestra la etiqueta "Requiere atención del estudio" en ROJO para
TODOS los mensajes donde needs_human_review = true. Esto confunde al dueño
porque ve mensajes de saludo ("Hola buen dia") en rojo, cuando en realidad
el sistema ya les respondió automáticamente y no hay nada que hacer.

La etiqueta roja debería aparecer SOLO cuando el sistema NO pudo responder
y genuinamente necesita intervención humana.

### Solución solicitada:

1. Agregar campo `bot_replied_at` (timestamp) a la tabla `inbound_messages`.
   Si es null → el bot NO respondió a este mensaje.
   Si tiene valor → el bot sí respondió automáticamente en esa fecha/hora.

   Migración sugerida:
   ```sql
   ALTER TABLE booking.inbound_messages
     ADD COLUMN IF NOT EXISTS bot_replied_at timestamptz DEFAULT NULL,
     ADD COLUMN IF NOT EXISTS outbound_message_id uuid REFERENCES booking.outbound_messages(id) DEFAULT NULL;
   ```

2. Cuando el bot envíe una respuesta automática a un mensaje inbound, el
   webhook/handler debe actualizar `bot_replied_at = now()` y
   `outbound_message_id` = el id del outbound_message generado.

3. El endpoint GET /api/owner/inbound-messages debe incluir estos campos en
   el response de cada mensaje:

   ```json
   {
     "id": "...",
     "botReplied": true,          // bot_replied_at IS NOT NULL
     "botRepliedAt": "2026-05-09T14:32:11Z",  // o null
     "outboundMessageId": "uuid o null",
     ... resto de campos existentes
   }
   ```

### Lógica de needs_human_review corregida:

La lógica actual debe refinarse. Propuesta:

  | intent       | confidence | bot_replied_at | needs_human_review correcto |
  |--------------|------------|----------------|-----------------------------|
  | greeting     | alta       | con valor      | FALSE                       |
  | thanks       | alta       | con valor      | FALSE                       |
  | hours        | alta       | con valor      | FALSE                       |
  | location     | alta       | con valor      | FALSE                       |
  | price        | alta       | con valor      | FALSE                       |
  | availability | alta       | con valor      | FALSE                       |
  | human_help   | cualquiera | null           | TRUE                        |
  | human_help   | cualquiera | con valor      | FALSE (bot ya respondió)    |
  | unknown      | < 0.5      | null           | TRUE                        |
  | booking      | cualquiera | null           | TRUE (cita no procesada)    |
  | reschedule   | cualquiera | null           | TRUE (cambio no procesado)  |
  | cancel       | cualquiera | null           | TRUE (cancelación pendiente)|

En resumen: needs_human_review = true SOLO cuando:
  - El bot no pudo responder (bot_replied_at IS NULL) Y
  - El intent requiere acción humana (no es greeting/thanks/hours/location/price)
